# Skill 实现策略：确定性流程用程序，不确定才用 LLM

## 规则

**能用程序化实现的 skill / 功能，不用 LLM 来实现。**

## 判断标准

| 特征 | 判定 | 实现策略 |
|---|---|---|
| 输入输出规则可穷举 | 确定性流程 | 代码模块（工具） |
| 逻辑分支有限、结果可验证 | 确定性流程 | 代码模块（工具） |
| 需要理解、语义判断、上下文推断 | 非确定性流程 | Agent skill（LLM） |
| 输出随上下文变化、逻辑不可穷举 | 非确定性流程 | Agent skill（LLM） |

## 原因

1. **省 token** — 程序调用零推理成本，LLM 每次调用消耗 token
2. **无幻觉** — 程序输出由逻辑决定，LLM 有概率性漂移
3. **成功率高** — 程序无随机性，LLM 有失败概率、需重试

## 适用范围

所有 skill 设计决策 / agent 功能拆分 / 新功能选型时的首选判据。

适用 agent：architect（skill 设计）/ coder（实装选型）/ director（审查设计合理性）

## 自检问题

> 「这个流程，能不能用代码把所有分支写完？能 → 写代码；不能 → 才考虑 LLM。」

## 反例（不应用 LLM 的场景）

- 文件路径拼接、格式校验、状态流转（枚举 → 枚举）
- channel 文件 frontmatter 解析和更新
- 模块依赖图遍历

## 正例（应用 LLM 的场景）

- 根据需求描述判断应该设计哪个模块
- 代码 review 的批判性分析
- session 内容的模式提炼与中期记忆压缩
