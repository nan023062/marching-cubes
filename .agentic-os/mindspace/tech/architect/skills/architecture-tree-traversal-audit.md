# 架构审查 — 树遍历四序方法

> **何时用**：架构师做全局/子树架构合规审查时。把数据结构里的"树遍历访问时机"映射到模块审查上，**4 种序回答 4 类不同的问题**，单序遗漏 2/3 的真相。
>
> **不适用**：单模块的细节代码 review（用 reviewer subagent）/ 单次 task 验收（按 task 验收 checklist）/ 漂移登记 append（按 append-only 戒律直接写）。

---

## 核心原则

**每种序的"访问时机"决定了它能回答什么问题**。混用 = 抓不到根因。

| 序 | 访问时机 | 在审查中的语义 | 回答的核心问题 |
|---|---|---|---|
| **层序（BFS）** | 按层逐节点 | 横向清点 + 评级 | 哪些节点健康 / 哪些不健康？ |
| **前序（Pre-order）** | 先父后子 | 自顶向下传递约束 | 父对子的"承诺/约束"是否被子兑现？ |
| **中序（In-order）** | 子—父—子（兄弟之间穿插） | 横向看兄弟之间的边界 | 兄弟之间是否正交无重叠？ |
| **后序（Post-order）** | 先子后父 | 自底向上聚合事实 | 父的"涌现性洞察"是否真的从子涌现？ |

> **N 叉树的中序定义**：「在访问每个子之间穿插看父」——本质是逐对兄弟做边界对照。

---

## 4 序的能查清单（实证）

### 层序（BFS）— 清点 + 评级

**访问规则**：按知识树层级（L0 根 → L1 → L2 → L3）逐节点扫，每个节点按 8 项打分（元数据 / 知识完整 / 定位 / 边界 / 依赖真实 / 戒律合规 / 内部组织 / 关系清晰），给 ✅/⚠/🚨 评级。

**能产出**：

- 全模块健康度地图（哪些节点健康 / 哪些🚨）
- 共性问题（如"所有 L1 模块都缺 boundary 字段"）
- 治理优先级排序
- inventory（清点）

**不能产出**：因果追溯 / 偏差根因。

**适用时机**：

- 接手陌生项目第一天的全景扫
- 季度治理回顾
- "全局审查 X 是否合理"类宽问题

### 前序（Pre-order）— 自顶向下查约束传递

**访问规则**：先 inspect 父，记录父对"子应该是什么样"的承诺；下钻每个子验证子是否兑现父的承诺。

**能查 6 类问题**：

| 检查项 | 例子 |
|---|---|
| 子模块清单完整性 | 父表格列了 N 个子，实际目录是 M 个子（漏报/虚报） |
| 父 constraints 的子继承 | 父声明"任何模块禁止直接 fs::*"——下钻 grep 子代码验证 |
| MKO 违反检测 | 父是否下沉了不该下沉的子细节（出现 `commands.rs:225` 这种文件行号 = MKO 违反） |
| 命名约定下沉 | 根级定的 kebab-case，下钻每个子模块名是否符合 |
| boundary 语义一致性 | 父声明子是 closed，下钻看子的 publicApi 是否真的为 0 |
| 跨子协议真相源 | 父 contract.md 列的"跨子协议"，下钻看子是否引用同一真相源 |

**适用时机**：

- 新模块刚立 → 确保子的设计承接父的承诺
- 架构师做 R+1 设计 → 确认新设计不破坏父的承诺
- manager 复盘合规审计

### 中序（In-order）— 横向查兄弟边界正交

**访问规则**：N 叉树的中序定义为"在访问每个子之间穿插看父"——本质是**逐对兄弟做边界对照**。

**能查 6 类问题**：

| 检查项 | 例子 |
|---|---|
| 兄弟职责正交性 | channel vs scheduler / log vs channel / ui vs ui-shell —— 任一对兄弟是否在做对方的事 |
| 横切依赖合规性 | 兄弟之间的依赖箭头方向是否符合六层调用链 |
| DIP 倒置正确性 | 兄弟间的反向依赖是否在 module.json 中正确声明（如 agent-plugins → kernel 是 Association） |
| 依赖图循环检测 | A→B→C→A 在中序遍历过程中能撞上 |
| 兄弟接口对齐 | channel 的 `Message` schema 字段 vs scheduler 消费的字段是否对得上 |
| 跨兄弟数据流真相源 | 同一个 DTO 在兄弟 A 定义、兄弟 B 消费——schema 真相源是哪一方 |

**适用时机**：

- 新增子模块前 → 对所有现有兄弟做边界对照（防新模块抢老模块职责）
- 依赖图 PR review
- 数据流设计

### 后序（Post-order）— 自底向上查事实涌现

**访问规则**：先彻底走完所有子（含子的实装代码），再回到父，验证父的"涌现性洞察 / 跨子协议 / 抽象封装"是否**真的是从子的事实涌现**，而非凭空想象。

**叶子模块的后序变体**：从最具体的事实（代码行）回溯到最抽象的承诺（祖父引用），按 L0 代码 → L1 contract → L2 architecture → L3 module.json → L4 父祖引用 顺序爬。

**能查 7 类问题**：

| 检查项 | 例子 |
|---|---|
| 涌现性洞察真伪 | 祖父说"六层调用链 + ipc-bridge 单点解耦"——遍历所有子代码验证是否真的没跳层 |
| 父 publicApi vs 子 publicApi 关系 | cbim 门面 55 方法 = crud 子模块方法的转发？还是独立加工？ |
| 跨子协议的真实接口集 | 父 contract 列的协议 = 子 contract 实际接口的并集 / 子集 / 不一致？ |
| 戒律真实贯彻度 | KPD/MKO/CBI 戒律——先扫子的实际行为，再回到父审"戒律宣告"是否对应 |
| 漂移自下而上的涌现 | 子层涌现的漂移——父祖的"洞察 / 待办"是否承接 |
| 抽象的有效性 | trait 设计是否真的覆盖了子的实装需求 |
| MVP 实际能力 vs 父声称范围 | 后序遍历所有子的实装行数 / 测试覆盖 → 回到祖父 § MVP 范围核对 |

**适用时机**：

- MVP 验收 / Demo 准备
- 架构 R+1 重设计前
- 复盘 lessons（漂移本质是"子事实 ≠ 父承诺"，后序最容易抓）
- cutover 完成后的"广播 sweep"

## 决策矩阵 — 何时用哪个

| 场景 | 推荐序 | 原因 |
|---|---|---|
| 全局清点 + 评级 | **BFS** | inventory，不追因果 |
| "全局审查 X 是否合理"宽问题 | **BFS 起步** + 后序深入 1-2 个🚨节点 | 先地图后挖矿 |
| 新模块入库前合规审计 | **前序** | 验证"承接父承诺" |
| 模块拆分 / 合并决策 | **中序** | 验证"兄弟边界正交" |
| MVP 验收 / 重大里程碑 | **后序** | 验证"父声称 vs 子事实" |
| 漂移登记入账 | **后序** | 漂移本质是"子事实 ≠ 父承诺" |
| 派新 task | **前序 + 中序** | 先确认新 task 不破坏父承诺，再看不抢兄弟职责 |
| 复盘提炼 lessons | **后序** | 从子层事实倒推父层盲点 |
| cutover 完成后的同步检查 | **后序** | 把 L0 事实广播到 L1-L4 |

---

## 误区（不能混用的反模式）

| 反模式 | 错在哪 |
|---|---|
| 用前序去查"事实" | 凭空指控子违反父的承诺，但子可能是对的、父过期 |
| 用后序去查"约束" | 看完子才知道父说了啥，太晚 |
| 用 BFS 找根因 | BFS 只能给评级，因果追溯需要 DFS 三序 |
| 只用一种序就出审查报告 | 单序遗漏 2/3 的真相 |

## 实证案例（2026-04-26 ipc-bridge 后序）

**任务**：审查 ipc-bridge 模块整体合理性。

**方法**：叶子模块的纵向后序（L0 代码 → L1 contract → L2 architecture → L3 module.json → L4 父祖引用）。

**结果**：

| 层 | 偏差数 | 性质 |
|---|---|---|
| L0 代码 | 0 | 完全自洽 |
| L1 叶子契约 | 3 | 双轨过渡 / 设计-实装混杂 / icon 注释 stale |
| L2 叶子架构 | 6 | 5 处 "5/4 commands/DTO" stale + 1 处自定阈值未触发评估 |
| L3 叶子元数据 | 3 | dependencies 漏 3 / publicApi stale / constraints 引用不存在的 PtyState |
| L4 父祖引用 | 5 | 父契约 stale / 父架构 stale 等 |
| L4 兄弟引用 | 3 | ui/architecture.md 对 ipc-bridge 的 D1/D2 描述 stale |

**总偏差 = 20 处**，全部由 **T6a cutover 后未做"反向广播同步"** 引起。

**后序独有的 5 个洞察**（前序/中序/BFS 无法发现）：

1. **L0 完美但 L4 严重过期 → 治理是"广播链路"问题，不是"代码质量"问题**
2. **偏差数量 = `L4 - L0`，与 cutover 涉及的依赖广度成正比**（ipc-bridge 涉 4 上游 → 8 处 L4 偏差）
3. **自定阈值未触发评估**（architect 在 architecture.md 写"DTO > 15 触发 ts-rs 评估"，实际已 17 但无机制提醒）
4. **append-only 戒律与"广播同步"职责的冲突**（worker 守 append-only 但导致末尾段越来越长，首读者被前文 stale 反复强化错觉）
5. **L1 双轨表述本质是 "designed vs implemented" 状态字段缺失**

## 派生治理准则（从后序洞察提炼）

> 这些是后序遍历**独有的**派生准则——前序/中序/BFS 都不会触发这些。

### 准则 1：cutover 任务的"广播工作量"应在 task 设计阶段估算

涉及 N 个上游模块的 cutover，**预期 L4 父祖偏差 ≈ 2N + 3**（每个上游对应 1 个 contract 引用 + 1 个 architecture 引用，加根级 stale）。task 验收 checklist 应含"广播 sweep"项。

### 准则 2：append-only 段超过一定篇幅 → architect 必须做合并重写

worker 守 append-only 是合规，但形成"原段 stale + 末尾 append 纠正"双轨。**architect 在合规验收同 commit 内做合并重写**（划线原段 + 注 commit hash + 升格 append 段为正式表格修订）。这职责不能下放 worker。

### 准则 3：自定阈值需要自动 reminder 机制

architecture.md 中的"自动评估阈值"陈述（如"DTO > 15"、"模块数 > 10"、"依赖深度 > 5"）需要自动巡检——否则等同摆设。建议 manager 巡检脚本 grep 这类陈述生成 reminder。

### 准则 4：contract.md 的 command/DTO 增加 status 字段

`status: implemented | designed | planned` —— 消除"实装段 vs 设计登记段"的视觉混杂，cbim/structure 模块层加约束。

### 准则 5：每次 cutover 后做一次叶子模块的纵向后序

5 层全扫（L0 → L4），关闭"反向广播缺失"漏洞。比"前序找漏"更有效，因为 cutover 的偏差总是从 L0 向 L4 扩散，后序顺向追踪。
